## 3. 环境性质与写操作授权
- 环境：**测试/预发，允许写操作**（下单、兑换券码、发货、增删商品/供货商/卡券均可执行）。
- 写操作要求**留痕可识别**：所有由验证产生的数据，命名统一加前缀 `E2E_` + 日期（如商品名 `E2E_20260627_测试商品`、收货人 `E2E_测试`），便于事后清理与区分。
- 兑换券码：消耗真实券码会改变券状态。优先用后台「卡券生成」自建一批 `E2E_` 批次券用于兑换测试，避免污染业务券。
- 破坏性/不可逆操作（取消订单、作废券、删除商品/供货商）：**仅在自建的 E2E_ 数据上执行**，绝不操作真实业务数据。

## 4. 可测性分级（决定每个用例怎么测）
- 🟢 **可自动化**：Playwright 在 PC后台 或 H5 上可完整走通。
- 🟡 **部分可测**：主流程可测，但依赖微信能力的环节（如手机号一键授权弹窗）需人工辅助或只验证到弹窗前。
- 🔴 **不可自动化**：微信小程序专有能力（微信一键登录、订阅消息授权、服务通知、微信客服），Playwright 无法覆盖 → 转为「人工核查 + 通过后台/代码侧面佐证」。

## 5. 工具
- 浏览器自动化：**Playwright MCP**（本工程已配置，输出在 `.playwright-mcp/`）。在本工程目录运行 claude 时可用。
- 截图/控制台日志：每个用例失败时务必抓 1) 截图 2) 控制台报错 3) 关键网络请求响应，存入 `.playwright-mcp/` 并在 FINDINGS 里引用路径。

## 6. 代码侧佐证（白盒辅助）
本工程是用户端 uniapp 源码，可用于判断「前端是否实现了某功能」：
- 页面路由：`pages.json`、`pages/`
- 插件能力：`pages/plugins/`（`coupon` 礼品券/卡券、`wallet` 余额、`shop` 多商户等）
- API 构造：`App.vue → get_request_url()`
当 Playwright 现象不明确时，可读源码确认「是前端没做 / 后端没返回 / 仅环境问题」，写进 FINDINGS 的「定位」字段，方便后端工程师。
